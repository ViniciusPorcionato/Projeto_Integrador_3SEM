Dois cenários, uma solução
Arquitetura em dois cenários
IaaS vs PaaS na prática
Visão geral

A XYZ quer avaliar duas abordagens diferentes de solução em nuvem. Todos os grupos devem implementar ambos os cenários, para que a empresa possa comparar vantagens, desvantagens e complexidade de cada um.

Núcleo comum: Em ambos os cenários, existe um CLP que envia dados (temperatura, botão) e recebe comandos (LED) via MQTT. A diferença está em como esses dados são processados e armazenados na nuvem.
☁️ Cenário 1 – Infraestrutura própria (IaaS)

Stack: EC2 + Mosquitto + Node-RED + MySQL

O que fazer:

  • Criar uma instância EC2 com Linux na AWS, que funcionará como "servidor industrial na nuvem".
  • Instalar e configurar dentro da EC2:
    • Mosquitto como broker MQTT;
    • Node-RED para orquestração de fluxos IoT;
    • MySQL para armazenamento de dados.

Fluxo de dados:

  1. CLP/simulador envia temperatura e evento de botão via MQTT.
  2. Mensagens chegam ao Mosquitto (broker na EC2).
  3. Node-RED assina os tópicos MQTT e:
    • recebe as mensagens;
    • processa os dados (ex.: aplica limites de temperatura, classifica estado em normal/atenção/crítico);
    • grava os dados no MySQL;
    • atualiza um dashboard web em tempo real.
  4. Dashboard exibe:
    • temperatura atual;
    • histórico em gráfico ou tabela;
    • registro visual de acionamento de botão (últimos eventos).
  5. Quando o supervisor/operador envia um comando via dashboard/app (ex.: "Acionar LED de alerta"):
    • mensagem vai para Node-RED;
    • Node-RED publica no Mosquitto;
    • Mosquitto encaminha ao CLP;
    • CLP aciona o LED.

Características:

  • Controle total: você gerencia tudo (SO, portas, segurança, backups).
  • Flexibilidade: pode customizar qualquer parte do fluxo.
  • Responsabilidade: você é responsável por manutenção, atualizações, escalabilidade.
  • DevOps: oportunidade de praticar provisionamento, versionamento, CI/CD.

Interface industrial:

  • Dashboard do Node-RED (web responsivo, acessível via navegador no celular).
  • Cores claras para estados (verde = normal, amarelo = atenção, vermelho = crítico).
  • Informações essenciais em destaque, sem poluição visual.
🌐 Cenário 2 – Serviços gerenciados (PaaS)

Stack: AWS IoT Core + BD gerenciado + MQTT Dash

O que fazer:

  • Configurar o AWS IoT Core como broker MQTT gerenciado:
    • criar "things" (representação do CLP);
    • gerar certificados e políticas de segurança;
    • definir tópicos MQTT para:
      • temperatura;
      • eventos de botão;
      • comandos para LED.
  • Criar IoT Rules (ou funções Lambda) para:
    • receber mensagens do broker;
    • armazenar em um banco de dados gerenciado (RDS ou DynamoDB).
  • Configurar o aplicativo MQTT Dash em um dispositivo móvel para:
    • conectar ao AWS IoT Core (via certificado);
    • assinar tópicos MQTT e exibir dados em tempo real;
    • publicar comandos (ex.: ligar/desligar LED).

Fluxo de dados:

  1. CLP/simulador envia temperatura e evento de botão via MQTT.
  2. Mensagens chegam ao AWS IoT Core (broker gerenciado).
  3. IoT Rules (ou Lambda) processam e armazenam os dados em um banco gerenciado.
  4. MQTT Dash no celular:
    • conecta ao AWS IoT Core;
    • recebe temperatura em tempo real (exibe em gauge, gráfico, etc.);
    • mostra indicador de eventos de botão;
    • permite enviar comando para ligar/desligar LED (publica em tópico específico).
  5. Quando um comando é publicado:
    • AWS IoT Core recebe;
    • encaminha ao CLP;
    • CLP aciona o LED.

Características:

  • Menos infraestrutura: AWS gerencia broker, escalabilidade, backups.
  • Menos responsabilidade: você foca em lógica, não em manutenção de servidor.
  • Segurança gerenciada: certificados, políticas, criptografia já vêm prontos.
  • Escalabilidade automática: suporta muitos dispositivos sem você se preocupar.
  • Custo variável: você paga por mensagens e armazenamento, não por servidor fixo.

Interface mobile:

  • MQTT Dash é um app genérico, mas você configura widgets claros e compreensíveis.
  • Nomes de tópicos e labels devem ser simples (ex.: "Temp Motor", "Evento Operador", "LED Manutenção").
  • Fácil de usar, mesmo para operadores sem experiência técnica.
Comparação: Cenário 1 vs Cenário 2
Aspecto Cenário 1 (EC2) Cenário 2 (PaaS)
Controle Total Parcial (AWS gerencia)
Responsabilidade Alta (manutenção, segurança) Baixa (AWS cuida)
Escalabilidade Manual (você gerencia) Automática
Custo Fixo (EC2 rodando 24/7) Variável (por uso)
Complexidade Maior (mais componentes) Menor (serviços prontos)
DevOps Mais prático (provisionamento) Mais conceitual (IaC)
Aprendizado Profundo (entender tudo) Focado (integração)
